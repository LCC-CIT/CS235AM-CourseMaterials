Application Architecture

Android App Architecture

  • Components- apps are made up of components. There are four kinds of components.
    • One type that every app must have: 
      • Activities
        • Each screen is managed by it’s own activity
        • A multi-screen app will have multiple activities
    • And three optional components:
      • Services: long-running background operations
      • Content providers: manage shared data
      • Broadcast receivers: respond to system-wide broadcast announcements
    • Intent objects are used to activate activities, services, and broadcast receivers
    • Every app must have a manifest file

Activity class

  • Provides a user interface (UI)
  • Contains a window in which to add UI elements
  • Inherits from the abstract Context class
  • Each activity in a app is independent of the others
  • Can be activated from within it’s own app or by another app

Loosely Coupled Activities

  • Typical desktop apps have a single entry point. In an Android app, any activity (screen) can be an entry point.
  • Similar to web applications. You can typically directly open any web page in a site.You can also directly start any activity in any app.

  • Context class

    • Android Developers' Context Class Reference
    • Provides access to the application's environment
    • Provides a means to access other app components
    • Used for:
      • Accessing Android services
      • Accessing preferences (settings) and permissions
      • Creating new views (controls)
      • Accessing device resources

  • Intent class

      Objects of this class can contain a message
    • It specifies an action for the OS or an activity to perform
      • Most often used to launch new activities
    • It carries data

  • AndroidManifest.xml

    • Required for every application
    • Registers information about the app:
      • Components that make up the application (including Activities)
      • Required permissions
      • Minimum OS version supported
      • MainLauncher activity (C# attribute on the Activity class, get's put into the manifest when compiled)
  • Activity Lifecycle


    • Using Intents to launch Activities

    Add an activity class to the app

    • As with the main activity, additional activities will be derived from the Activity class.
    • Additional activities should not have the MainLauncher attribute.
    • Each additional activity will have it's own UI
    • Caution, the Xamarin Activity class has an Intent property-- so there is both a property and class named Intent

    Send an intent to start an activity

    Assume our app starts an activity named MainActivity and we have an additional activity named SecondActivity. In MainActivity, we will do this:

    1. Create an Intent object
    2. Add data to the intent (optional)
    3. Start the activity
    var myIntent = new Intent(this, typeof(SecondActivity));
    myIntent.PutExtra("Answer", 42);
    StartActivity(myIntent);

    Retrieve information from an Intent

    In SecondActivity, use the Intent property to get the data sent via the intent object.

    int answer = Intent.Extras.GetInt ("Answer");

    Getting a response from an Activity

    In MainActivity (the activity sending the intent):

    var myIntent = new Intent(this, typeof(SecondActivity));
    myIntent.PutExtra("Answer", 42);
    StartActivityForResult(myIntent, 0);

    In SecondActivity (the activity receiving the intent and sending back info):

    var myIntent = new Intent(this, typeof(MainActivity));  myIntent.PutExtra("RightAnswer", true); SetResult(Result.Ok, myIntent);
    Finish();
